# Локальный broker

## Назначение

Broker — это локальный WebSocket-сервер, через который браузерное расширение и Pi-сессии обмениваются служебными сообщениями Browser Connect.

Текущие параметры по умолчанию:

- **host:** `127.0.0.1`
- **port:** `17345`
- **URL:** `ws://127.0.0.1:17345`

Значения задаются в `src/shared/constants.ts`.

## Кто владеет брокером

Брокер поднимается не отдельным демоном, а одной из Pi-сессий после команды `/browser-connect`.

Алгоритм текущего MVP:

1. Pi-сессия сначала пытается подключиться к уже существующему брокеру.
2. Если подключение не удалось, она пытается поднять собственный broker server на фиксированном порту.
3. Если старт брокера проиграл гонку за порт (`EADDRINUSE`), код считает, что другой процесс уже успел поднять сервер, и повторяет обычное подключение к нему.
4. После успешной регистрации текущая Pi-сессия становится обычной целью. Если именно она подняла сервер, она ещё и владеет жизненным циклом брокера.

Это делает старт устойчивее при нескольких одновременных `/browser-connect`, но не превращает брокер в независимую службу.

## Аутентификация

### Общий токен

Broker принимает соединения только с общим токеном, который Pi-часть читает или создаёт в файле:

```text
.pi/browser-connect.token
```

Токен:

- создаётся как UUID;
- переиспользуется при следующих запусках в том же рабочем каталоге;
- хранится в директории `.pi` с правами `0700`;
- сам файл токена принудительно защищается правами `0600`.

### Аутентификация браузерного клиента

Для браузерных запросов есть два важных правила:

1. `client.listTargets` разрешён только **после успешного `client.hello`**.
2. `client.sendSelection` дополнительно несёт токен в собственном payload.

Если токен неверный или клиент пытается перечислять цели без аутентификации:

- broker отправляет `client.error`;
- после этого закрывает соединение.

На практике background-скрипт расширения сейчас делает `client.hello` и перед `client.listTargets`, и перед `client.sendSelection`.

### Аутентификация Pi-цели

Pi-сессия регистрируется через `target.register` и тоже обязана передать валидный токен. При неверном токене broker просто разрывает соединение.

## Что broker хранит в памяти

Для каждой активной Pi-цели broker держит:

- `targetId`
- `alias`
- `cwd`
- `gitBranch`
- `pid`
- `sessionName`
- `connectedAt`
- `lastSeenAt`
- WebSocket-соединение цели

Список целей не сохраняется на диск и живёт только в памяти текущего broker process.

## Жизненный цикл цели

### Регистрация

После `target.register` broker:

- сохраняет метаданные цели;
- обновляет `lastSeenAt` текущим временем;
- привязывает `targetId` к socket;
- подтверждает регистрацию сообщением `target.registered`.

Если тот же `targetId` зарегистрирован повторно, старое соединение вытесняется новым.

### Heartbeat и stale cleanup

Pi-цель отправляет heartbeat каждые `5_000 ms`.

Broker считает цель stale, если heartbeat не обновлялся `15_000 ms`, после чего:

- удаляет цель из списка;
- закрывает её socket;
- завершает связанные pending deliveries ошибкой.

### Явное отключение

При `target.unregister` broker удаляет цель и закрывает socket.

## Доставка выделения

Когда браузер вызывает `client.sendSelection`, broker:

1. проверяет токен и payload;
2. находит `targetId`;
3. создаёт внутренний `requestId` для доставки;
4. отправляет цели `target.deliverSelection`;
5. ждёт `target.sendSelectionResult`.

Таймаут доставки по умолчанию — `30_000 ms`.

Если цель:

- не найдена — браузер получает `Target is not available`;
- не ответила вовремя — браузер получает `Delivery timed out`;
- отключилась — pending delivery завершается ошибкой отключения.

## Логи и наблюдаемость

Pi-часть пишет события брокера и цели в:

```text
.pi/browser-connect.log
```

Логи помогают понять:

- кто поднял broker server;
- когда зарегистрировалась или отключилась цель;
- были ли ошибки сокета, таймауты или гонка старта.

## Известные ограничения MVP

- Broker слушает только localhost и не поддерживает удалённые подключения.
- Нет постоянного процесса-супервизора: broker живёт вместе с owning Pi-сессией.
- Нет очереди сообщений и повторной доставки после рестарта.
- Список целей раскрывает локальные метаданные (`cwd`, branch, PID, sessionName`) любому клиенту, который знает токен.
- Из-за фиксированного порта `17345` и токена, который берётся из `.pi/browser-connect.token` текущего каталога, несколько независимых проектов на одной машине могут конфликтовать между собой. Практически это означает, что в MVP безопаснее считать broker «привязанным» к одному проектному токену за раз.
