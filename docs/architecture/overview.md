# Обзор архитектуры

## Что это такое

Pi Chrome Assistent — это локальная связка из трёх частей:

1. **Pi-расширение** — даёт команды `/chrome-assistent-auth` и `/chrome-assistent-connect [alias]`.
2. **Локальный broker** — WebSocket-сервер на `ws://127.0.0.1:17345`.
3. **Chrome MV3-расширение** — popup, background service worker и content script.

Схема обмена:

```text
Chrome popup / background / content script
                  │
                  ▼
        ws://127.0.0.1:17345
                  │
                  ▼
         Локальный browser broker
                  │
                  ▼
        Pi session A / B / C с /chrome-assistent-connect
```

MVP отправляет данные **из браузера в Pi**. Ответ Pi остаётся в терминале, а браузер получает только статус доставки и диагностику.

## Основные роли компонентов

### Pi-расширение

- хранит общий runtime в `~/.pi/chrome-assistent/`;
- ведёт глобальный служебный токен брокера;
- ведёт список доверенных браузеров;
- командой `/chrome-assistent-auth` добавляет токен браузера в доверенный реестр;
- командой `/chrome-assistent-connect [alias]` публикует текущую Pi-сессию как цель.

### Локальный broker

- принимает регистрации Pi-целей по внутреннему target token;
- принимает браузерные запросы по browser token;
- возвращает список активных Pi-целей;
- маршрутизирует `client.sendSelection` в выбранную цель;
- очищает stale-цели по heartbeat.

### Chrome-расширение

Popup разделён на три вкладки:

- **Асистент** — только основной action **«Отправить в Pi»**;
- **Сессии** — список целей Pi и диагностика;
- **Авторизация** — токен браузера, копирование, перевыпуск и удаление.

Background работает посредником между popup/content script и broker, а content script отвечает за DOM picker и сбор полезного фрагмента страницы.

## Пользовательский поток: первое подключение

1. Пользователь запускает `pi` в нужном проекте.
2. При необходимости выполняет `/reload`.
3. В popup открывает вкладку **«Авторизация»** и копирует токен браузера.
4. В Pi выполняет `/chrome-assistent-auth` и вставляет этот токен.
5. В нужной Pi-сессии выполняет `/chrome-assistent-connect [alias]`.
6. В popup на вкладке **«Сессии»** выбирает нужную цель.
7. На вкладке **«Асистент»** запускает отправку в Pi.

Подробности: [Pi-расширение](./pi-extension.md), [broker](./broker.md), [авторизация браузера](../operations/token-setup.md).

## Пользовательский поток: отправка фрагмента страницы

1. Пользователь открывает вкладку **«Асистент»**.
2. Background уже знает выбранную Pi-цель из вкладки **«Сессии»**.
3. При нажатии **«Отправить в Pi»** background внедряет `contentScript.js` в активную вкладку.
4. Content script запускает DOM picker.
5. После подтверждения собирается payload:
   - `url`, `title`;
   - `selectedText`;
   - `selectedHtml`;
   - `selector`;
   - `comment`;
   - `capturedAt`.
6. Background проходит browser-auth через `client.hello` и отправляет `client.sendSelection`.
7. Broker доставляет payload выбранной цели Pi.
8. Pi форматирует сообщение и вызывает `pi.sendUserMessage(...)`.

## Какие данные реально передаются

В Pi уходит текстовое сообщение с метаданными страницы и двумя основными блоками:

- выбранный текст — с ограничением `50 KiB`;
- HTML-фрагмент — с ограничением `100 KiB`.

Ограничения заданы в `src/shared/constants.ts`.

## Что важно помнить про текущую версию

- Broker слушает только `127.0.0.1`.
- Доверие разделено на два домена: внутренний target token для Pi и browser token для расширения.
- Ответы Pi обратно в браузер не стримятся.
- Жизненный цикл broker по-прежнему зависит от активной Pi-сессии.
