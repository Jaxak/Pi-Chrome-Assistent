# Pi-расширение

## Что делает команда `/chrome-assistent-connect`

Pi-часть регистрирует команду:

```text
/chrome-assistent-connect [alias]
```

Команда подключает текущую Pi-сессию к локальному broker-контру Browser Connect и делает эту сессию доступной для выбора из popup браузерного расширения.

Необязательный `alias` попадает в метаданные цели и используется как основной ярлык в списке целей.

## Где Pi хранит служебные файлы

### Общий токен

Файл токена создаётся или читается по пути:

```text
.pi/browser-connect.token
```

Особенности реализации:

- директория `.pi` создаётся с правами `0700`;
- файл токена защищается правами `0600`;
- симлинки для каталога и файла токена запрещены;
- если файл уже существует и содержит токен, он переиспользуется.

### Логи

Pi-логи Browser Connect пишутся в:

```text
.pi/browser-connect.log
```

## Какие метаданные регистрируются

При подключении Pi строит объект `TargetMetadata`:

- `targetId` — UUID текущей цели;
- `alias` — пользовательский псевдоним из `/chrome-assistent-connect [alias]`, если указан;
- `cwd` — текущий рабочий каталог сессии;
- `gitBranch` — текущая git-ветка, если её удалось определить;
- `pid` — PID процесса Pi;
- `sessionName` — имя сессии Pi, если доступно API;
- `connectedAt` — время первого подключения;
- `lastSeenAt` — время последнего heartbeat.

Эти данные потом видны в popup после аутентифицированного `client.listTargets`.

## Поток старта и присоединения к broker

Текущий код в `src/pi/browserConnectExtension.ts` делает следующее:

1. Нормализует `alias`.
2. Читает или создаёт общий токен.
3. Закрывает предыдущую активную target-связь, если она была.
4. Собирает свежие метаданные цели.
5. Пытается зарегистрироваться в брокере на:
   - уже известном `ownedBroker.port`, если текущая сессия владеет broker server;
   - иначе на `127.0.0.1:17345`.
6. Если подключение не удалось:
   - при необходимости очищает устаревший `ownedBroker`;
   - пытается запустить новый broker server;
   - если запуск проиграл гонку за порт (`EADDRINUSE`), повторяет подключение к уже существующему серверу.
7. После успешной регистрации обновляет статус и уведомление в UI Pi.

## Что видит пользователь в Pi

После успешного подключения Pi показывает статус вида:

```text
/chrome-assistent-connect: <label> · подключено · 127.0.0.1:<port>
```

И уведомление:

```text
Подключение /chrome-assistent-connect активно: <label> · 127.0.0.1:<port>
```

При ошибке команда сообщает:

```text
Не удалось выполнить /chrome-assistent-connect: <текст ошибки>
```

При неожиданном отключении активной цели:

```text
Подключение /chrome-assistent-connect прервано: <label> · 127.0.0.1:<port>
```

## Как Pi получает выделение из браузера

После доставки через broker Pi вызывает `handleDeliveredSelection(...)`, который:

1. форматирует браузерный payload в текстовое сообщение для Pi;
2. проверяет, простаивает ли сейчас Pi;
3. вызывает `pi.sendUserMessage(...)`:
   - без дополнительных опций, если Pi idle;
   - с `deliverAs: "followUp"`, если Pi уже занят ответом.

Итог: браузер только отправляет контекст, а полноценный ответ пользователь читает в терминале Pi.

## Формат сообщения, которое попадает в Pi

Сообщение содержит:

- URL страницы;
- title;
- selector, если его удалось построить;
- комментарий пользователя;
- выбранный текст;
- HTML-фрагмент.

Текст и HTML проходят ограничение размера и оборачиваются в fenced code blocks.

## Завершение работы

Pi-расширение очищает состояние в двух случаях:

- при повторном `/chrome-assistent-connect` — старая target-связь закрывается перед новым подключением;
- при `session_shutdown` — закрывается target connection и, если нужно, owning broker.

Если активная target-связь обрывается неожиданно, статус в Pi сбрасывается.

## Известные ограничения MVP

- Ответы Pi не стримятся в popup и не возвращаются в браузер.
- Browser token на стороне Chrome надо настраивать вручную.
- Жизненный цикл broker server всё ещё связан с Pi-сессией, которая его подняла.
- Из-за пути токена `.pi/browser-connect.token` в текущем `cwd` несколько разных проектов на одной машине не образуют прозрачный общий пул целей: токены могут различаться, а порт у брокера один и тот же.
