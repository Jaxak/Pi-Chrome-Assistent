# Ручная настройка brokerToken

## Зачем это нужно

В текущем MVP браузерное расширение не умеет само получать токен от Pi. Поэтому `brokerToken` нужно вручную перенести из файла Pi в `chrome.storage.local`.

Без этого popup не даст начать отправку и покажет подсказку:

```text
Для отправки настройте brokerToken в chrome.storage.local.
```

## Шаг 1. Создайте токен на стороне Pi

В нужном проекте запустите Pi и выполните:

```text
/browser-connect [alias]
```

Если всё прошло штатно, в каталоге проекта появится файл:

```text
.pi/browser-connect.token
```

## Шаг 2. Скопируйте токен из файла

Пример через терминал:

```bash
cat .pi/browser-connect.token
```

Скопируйте значение без лишних пробелов и переносов.

## Шаг 3. Откройте DevTools расширения

Один из рабочих вариантов для Chrome:

1. Откройте `chrome://extensions`.
2. Найдите расширение **Pi Chrome Assistent**.
3. Откройте страницу деталей расширения.
4. Нажмите ссылку для service worker / background page DevTools.
5. Перейдите на вкладку **Console**.

## Шаг 4. Запишите токен в `chrome.storage.local`

В консоли DevTools выполните:

```js
await chrome.storage.local.set({
  brokerToken: "<токен из .pi/browser-connect.token>"
});
```

## Шаг 5. Проверьте, что значение сохранилось

```js
await chrome.storage.local.get("brokerToken")
```

Ожидаемый результат:

```js
{ brokerToken: "<тот же токен>" }
```

## Шаг 6. Обновите popup

Закройте и снова откройте popup расширения. После этого:

- сообщение про обязательный `brokerToken` должно исчезнуть;
- кнопка **Отправить в Pi** станет доступной, если есть активные цели и выбрана цель.

## Как сменить токен

Если вы переключились на другой broker-контур или другой проект, нужно вручную заменить значение:

```js
await chrome.storage.local.set({
  brokerToken: "<новый токен>"
});
```

## Как удалить токен

```js
await chrome.storage.local.remove("brokerToken")
```

После удаления popup снова будет считать отправку недоступной.

## Важное ограничение текущего MVP

Broker всегда пытается работать на фиксированном порту `17345`, а токен берётся из `.pi/browser-connect.token` текущего проекта. Поэтому одновременно удобно поддерживать только один согласованный токен на один локальный broker-контур.

Если вы запускаете `/browser-connect` в другом проекте и используете другой токен, может возникнуть рассинхронизация:

- Pi-сессия не сможет присоединиться к уже работающему broker с чужим токеном;
- popup будет видеть ошибки аутентификации или отсутствие целей;
- придётся вручную выровнять, какой именно проект и какой токен сейчас считаются «активными».
