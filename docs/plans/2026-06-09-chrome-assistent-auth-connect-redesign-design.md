# Дизайн: упрощённая авторизация браузера и подключение Pi-сессий

Дата: 2026-06-09
Статус: validated brainstorm

## Контекст и цель

Текущий UX подключения слишком технический: пользователю нужно собирать расширение, вручную переносить `brokerToken` из файлов Pi в `chrome.storage.local`, разбираться с DevTools service worker и диагностикой. Цель редизайна — привести сценарий к понятной продуктовой модели:

1. пользователь устанавливает расширение;
2. авторизует конкретную установку браузерного расширения;
3. отдельно подключает нужную Pi-сессию;
4. дальше работает через popup без ручного переноса токенов и без поиска нужного действия среди технических деталей.

Дополнительно принято решение переименовать команду `/browser-connect` в `/chrome-assistent-connect` **без обратной совместимости**, так как решением ещё никто не пользуется.

## Принятые решения

### 1. Авторизация и подключение сессии разделяются

Система делится на два независимых контура:

- **Авторизация браузера** — одноразовое доверие конкретной установке расширения.
- **Подключение Pi-сессии** — отдельное действие в нужной сессии Pi.

Команды:

- `/chrome-assistent-auth` — принимает токен браузера и добавляет его в глобальный список доверенных.
- `/chrome-assistent-connect [alias]` — публикует текущую Pi-сессию как доступную цель.

Авторизация **не** публикует текущую сессию автоматически.

### 2. Отправная точка авторизации — расширение

Вместо deep-link, локальных redirect-flow и автоматического открытия страниц авторизация начинается в браузерном расширении:

1. пользователь открывает расширение;
2. нажимает кнопку авторизации;
3. расширение генерирует токен конкретной установки;
4. пользователь копирует токен;
5. в Pi запускает `/chrome-assistent-auth` и вставляет токен.

Это менее магический, но более устойчивый сценарий. Он не зависит от стабильного `chrome-extension://` URL, одинаково работает для developer mode и для версии из магазина и проще в поддержке.

### 3. Каждая установка расширения — отдельное доверенное устройство

Токен генерируется локально в расширении и представляет **конкретную установку** браузерного расширения. Не используется общий токен на все браузеры пользователя.

Это даёт простую и понятную модель:

- можно авторизовать несколько браузеров независимо;
- можно перегенерировать токен только в одном браузере;
- можно отвязать конкретную установку, не ломая остальные.

### 4. Токен многоразовый

Токен не является одноразовым pairing-code. Он живёт, пока пользователь сам не выполнит одно из действий:

- **Скопировать токен**
- **Сгенерировать новый токен**
- **Отвязать браузер**

Это решение выбрано в пользу простоты UX. При перегенерации токена старая авторизация перестаёт работать, пока новый токен не будет заново подтверждён через `/chrome-assistent-auth`.

### 5. Хранилище доверенных токенов — глобальное

Pi хранит доверенные токены **глобально на уровне пользователя**, а не проекта. Логически это реестр доверенных браузеров, доступный всем локальным проектам и всем будущим сессиям Pi на этой машине.

Первая версия может хранить минимальную запись: токен и метаданные времени добавления. Но структура должна позволять позже развить решение до полноценного списка устройств.

## Потоки данных

### Поток A. Авторизация браузера

1. Пользователь открывает вкладку авторизации в popup.
2. Расширение читает токен из `chrome.storage.local`.
3. Если токена нет — генерирует новый.
4. Пользователь копирует токен.
5. В Pi запускает `/chrome-assistent-auth`.
6. Pi через интерактивный prompt запрашивает токен.
7. После успешной проверки формат/ввод токен добавляется в глобальный список доверенных.
8. С этого момента запросы расширения с этим токеном считаются авторизованными.

На этом этапе никакая Pi-сессия ещё не публикуется.

### Поток B. Подключение текущей Pi-сессии

1. Пользователь в нужной Pi-сессии запускает `/chrome-assistent-connect [alias]`.
2. Pi поднимает или находит локальный broker.
3. Текущая сессия регистрируется как target.
4. Расширение при открытии popup предъявляет свой токен.
5. Broker сначала проверяет, что токен доверенный.
6. Только после этого расширение может:
   - получить список доступных сессий;
   - отправить выделение в выбранную Pi-сессию.

Если токен не доверен, список рабочих целей не раскрывается.

## Требования к UI popup

Popup должен стать **примерно на 30% уже**, а основное действие пользователя должно быть вынесено на первый план. Технические детали не должны мешать ежедневному сценарию.

### Вкладки popup

#### 1. `Асистент`

Главная вкладка для повседневной работы.

Здесь размещаются:

- кнопки отправки в Pi;
- минимальный статус готовности;
- в будущем — дополнительные пользовательские сценарии взаимодействия с Pi.

На этой вкладке **не должно быть диагностического шума**.

#### 2. `Сессии`

Вкладка управления доступными Pi-сессиями.

Здесь размещаются:

- список опубликованных сессий;
- выбор активной сессии;
- alias и связанные данные;
- диагностическая информация и ошибки подключения.

Именно сюда выносится всё, что связано с `/chrome-assistent-connect` и состоянием broker.

#### 3. `Авторизация`

Вкладка управления токеном браузера.

Здесь размещаются:

- текущий статус авторизации;
- показ токена;
- кнопка копирования;
- перегенерация токена;
- отвязка браузера;
- краткая инструкция по использованию `/chrome-assistent-auth`.

## Ошибки и пользовательские статусы

Все пользовательские тексты должны быть на русском языке.

Минимальный набор состояний/сообщений:

- `Браузер не авторизован в Pi`
- `Нет подключённых сессий Pi`
- `Выбранная сессия больше недоступна`
- `Не удалось связаться с локальным сервисом Pi`
- `Токен не распознан`
- `Авторизация браузера была отозвана`

Диагностика должна быть доступна на вкладке `Сессии`, но не должна перегружать вкладку `Асистент`.

## Последствия для архитектуры

1. Убирается сценарий с ручной передачей `.pi/browser-connect.token` в `chrome.storage.local`.
2. Авторизационный токен генерируется и хранится на стороне браузерного расширения.
3. Broker должен уметь проверять токен браузера против глобального реестра доверенных токенов Pi.
4. Команда `/browser-connect` удаляется и заменяется на `/chrome-assistent-connect`.
5. Появляется новая команда `/chrome-assistent-auth`.
6. Документация и onboarding переписываются под новый двухшаговый сценарий: сначала доверие браузеру, потом публикация конкретной сессии.

## Что проверять при реализации

Минимальный набор проверок:

1. расширение генерирует и сохраняет токен локально;
2. `/chrome-assistent-auth` сохраняет токен в глобальном хранилище Pi;
3. неавторизованный токен не получает список сессий;
4. авторизованный токен получает список сессий;
5. `/chrome-assistent-connect` публикует текущую сессию;
6. после перегенерации токена старый токен перестаёт работать;
7. после отвязки браузер возвращается в неавторизованное состояние;
8. popup корректно разделяет действия по вкладкам `Асистент | Сессии | Авторизация`.

## Рекомендуемый следующий шаг

Следующий этап — составить детальный план реализации:

1. переименование команды и терминологии `browser-connect` → `chrome-assistent-connect`;
2. проектирование глобального хранилища доверенных токенов;
3. добавление `/chrome-assistent-auth`;
4. изменение протокола broker для проверки доверенных токенов;
5. переработка popup под новую информационную архитектуру;
6. обновление пользовательской документации и сценариев установки.
