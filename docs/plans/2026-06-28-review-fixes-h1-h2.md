## H1: Убрать `<all_urls>` из `host_permissions`

### Проблема

`manifest.json` содержит `"host_permissions": ["http://127.0.0.1/*", "ws://127.0.0.1/*", "<all_urls>"]`.  
`<all_urls>` даёт расширению программный доступ к содержимому любого сайта, что не используется в коде и создаёт лишний attack surface.

### Анализ использования

Все вызовы Chrome tabs API в коде:

| Файл | Вызов | Зачем |
|------|-------|-------|
| `sidepanel.ts:56` | `chrome.tabs.query({ active: true, currentWindow: true })` | Найти активную вкладку для DOM picker |
| `backgroundStateServer.ts:385` | `chrome.tabs.query({ active: true, currentWindow: true })` | Найти вкладку для stopDomPicker |
| `backgroundStateServer.ts:387` | `chrome.tabs.sendMessage(tab.id, ...)` | Отправить stopDomPicker в content script |

Все три вызова работают только с **активной вкладкой**. Разрешение `activeTab` (уже есть) + `tabs` покрывает эти сценарии. `<all_urls>` не требуется.

Content script объявлен в `content_scripts.matches` — ему `host_permissions` не нужен для инжекции.

### Исправление

**Файл:** `src/chrome/manifest.json`  
**Изменение:** удалить `"<all_urls>"` из массива `host_permissions`

**Было:**
```json
"host_permissions": ["http://127.0.0.1/*", "ws://127.0.0.1/*", "<all_urls>"]
```

**Стало:**
```json
"host_permissions": ["http://127.0.0.1/*", "ws://127.0.0.1/*"]
```

### Тесты

Изменение не затрагивает runtime-код. Проверить:

- [ ] Расширение загружается в Chrome без ошибок
- [ ] DOM picker работает на обычной HTTP/HTTPS странице
- [ ] Side panel подключается к Pi-сессии

### Сложность: тривиальная (1 строка)
### Время: 2 минуты

## H2: Убрать автоматический reconnect, перейти на ручное подключение

### Проблема

Сейчас `SessionClient` при потере соединения автоматически переподключается каждую секунду навсегда. Если Pi не запущен — расширение бессмысленно молотит `ws://127.0.0.1:31415` десятки тысяч раз в день.

Кроме того, при открытии sidepanel происходит автоматическое восстановление сохранённого порта (`tryRestoreSavedPort`) и автоматическое подключение — пользователь не просил подключаться.

### Целевое поведение

**Сценарий 1: Пользователь запускает Pi, потом открывает браузер**
1. Пользователь запускает Pi и активирует ассистента
2. Открывает браузер, sidepanel показывает "Pi не подключён. Введите порт и нажмите «Подключить»."
3. Пользователь нажимает «Подключить» → расширение подключается к Pi-сессии
4. Если в течение 60 минут нет активности (нет входящих сообщений от Pi) → расширение отключается, показывает "Сессия неактивна. Нажмите «Подключить» для восстановления."
5. Пользователь нажимает «Подключить» → переподключение

**Сценарий 2: Пользователь открывает браузер без Pi**
1. Пользователь открывает браузер, Pi не запущен
2. Sidepanel показывает "Pi не подключён. Введите порт и нажмите «Подключить»."
3. Никаких автоматических запросов к WebSocket
4. Пользователь нажимает «Подключить» → одна попытка подключения
5. Если Pi не найден → "Pi-сессия не найдена на порту 31415. Запустите Pi и нажмите «Подключить»."
6. Никаких автоматических повторных попыток

**Сценарий 3: Pi выключается во время работы**
1. Подключение было установлено, пользователь работал
2. Pi завершает сессию → WebSocket закрывается
3. Расширение делает ограниченный reconnect (3-5 попыток за ~5 секунд) — на случай кратковременного разрыва
4. Если reconnect не удался → "Соединение с Pi потеряно. Нажмите «Подключить» для восстановления."
5. Никаких дальнейших автоматических попыток

### Ключевое правило

> **Reconnect допускается только после успешного подключения**, и только ограниченное количество раз. Если соединение никогда не было установлено — ноль reconnect-попыток.

### Детальный план изменений

#### Задача 1: SessionClient — убрать бесконечный reconnect, добавить `everConnected`

**Файл:** `src/chrome/sessionClient.ts`

1. **Добавить флаг `everConnected`** — становится `true` после получения первого валидного snapshot'а.

2. **Изменить логику `scheduleReconnect()`:**
   - Если `everConnected === false` → **не** reconnect'ить. Вызвать `reportState(false, false, "Pi-сессия не найдена на порту {port}. Запустите Pi и нажмите «Подключить».")` и остановиться.
   - Если `everConnected === true` → reconnect с ограничением: максимум `MAX_RECONNECT_ATTEMPTS = 5` попыток (ramp: 500ms → 1s → 2s → 3s → 5s). После исчерпания → `reportState(false, false, "Соединение с Pi потеряно. Нажмите «Подключить» для восстановления.")` и остановиться.

3. **Изменить `onClose` handler в `doConnect()`:**
   - Текущее: `if (!this.closedByClient && this.socket === undefined) { ... this.scheduleReconnect(); }`
   - Новое: добавить проверку `everConnected` и `reconnectAttempt < MAX_RECONNECT_ATTEMPTS` перед вызовом `scheduleReconnect()`.

4. **Сбросить `everConnected` в `close()`** — при явном отключении клиентом, следующий `connect()` должен начать с чистого состояния.

5. **Сбросить `reconnectAttempt` и `everConnected` в `connect()` и `reconnectToPort()`** — каждое новое ручное подключение — свежий старт.

6. **Добавить idle timeout (5 минут):**
   - Новое поле `idleTimer: ReturnType<typeof setTimeout> | undefined`
   - Новая опция `idleTimeoutMs?: number` (default: `3_600_000` — 60 минут)
   - При получении **любого** сообщения от сервера (snapshot, event, command.result) → сбросить idle timer
   - Когда idle timer срабатывает → закрыть соединение с `reportState(false, false, "Сессия неактивна. Нажмите «Подключить» для восстановления.")`
   - При `close()` → очистить idle timer
   - Idle timer не должен включаться, если `everConnected === false`

**Константы:**
```typescript
const DEFAULT_RECONNECT_DELAYS_MS = [500, 1_000, 2_000, 3_000, 5_000];
const MAX_RECONNECT_ATTEMPTS = 5;
const DEFAULT_IDLE_TIMEOUT_MS = 60 * 60 * 1_000; // 60 минут
```

#### Задача 2: BackgroundStateServer — убрать автоподключение при восстановлении порта

**Файл:** `src/chrome/backgroundStateServer.ts`

1. **Изменить `tryRestoreSavedPort()`:**
   - Текущее: восстанавливает порт И вызывает `handleSessionConnect(savedPort)` → автоподключение.
   - Новое: восстанавливает порт только в `state.connection.configuredPort` через `applyState({ kind: "connection_updated", connection: { configuredPort: savedPort } })`. **Не подключается автоматически.**

2. **Убедиться, что `handleSessionConnect` по-прежнему работает по нажатию «Подключить»** — ничего не ломается, т.к. sidepanel шлёт `assistant.session.connect` при клике.
