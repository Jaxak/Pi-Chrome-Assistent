# 1. Executive summary

The branch is in generally good shape: the codebase is modular, the shared protocol/truncation logic is well-factored, the Pi-side token file handling is unusually careful for an MVP, and the automated test suite is broad for a first release. I verified `npm test`, `npm run typecheck`, `npm run build:chrome`, `pi --version`, and a temporary-`HOME` Pi smoke run of `/browser-connect review-smoke`; all succeeded in this environment.

That said, I do **not** think the implementation is ready to keep as-is. There are two blocking engineering issues and one additional important robustness gap:

- the broker exposes target enumeration without token authentication, which leaks cwd/branch/pid/session metadata to any local WebSocket client;
- the popup still lets the user enter the picker/send flow even when no broker token is configured, so the UX fails late after the user has already done the DOM selection work;
- concurrent broker startup is race-prone: if two Pi sessions try to become the broker at nearly the same time, the loser can fail the command instead of falling back to reconnecting.

# 2. Strengths

- Clear module boundaries between shared protocol, Pi broker/client, and Chrome UI/runtime code.
- Good amount of focused automated coverage: protocol, truncation, broker integration, target client lifecycle, popup behavior, DOM picker heuristics, and build output.
- Security-conscious token-file creation/hardening in `src/pi/browserConnectExtension.ts`.
- Logging intentionally avoids dumping full selection payloads by default.
- DOM selection flow is decomposed sensibly into picker, overlay, toast, and background messaging modules.
- Pi smoke path works in this environment: `/browser-connect` starts, registers, and shuts down cleanly.

# 3. Findings

## Critical

- None.

## Important

### 1) Unauthenticated clients can enumerate active Pi targets and read workspace metadata
- **Files:** `src/pi/broker.ts:394-416`, `src/chrome/background.ts:281-297`, `src/pi/broker.test.ts:238-274`
- **Problem:** `client.hello` checks the token, but the broker does not require authentication state for `client.listTargets`. The Chrome background client also never sends `client.hello` before listing. As implemented, any socket that can reach `ws://127.0.0.1:17345` can call `client.listTargets` and receive aliases, cwd paths, git branches, PIDs, and session names.
- **Why it matters:** This is a real privacy leak, not just a protocol nicety. Browser pages can generally open WebSockets to localhost, so a visited site can fingerprint the developer's local projects/branches/processes even without the shared token. The current tests explicitly lock in this unauthenticated behavior.
- **Review recommendation:** Require authenticated session state for all client operations that expose broker data, including `client.listTargets`; make the Chrome background perform the handshake (or include the token on list requests), and replace the current broker test that asserts listing works “without prior hello” with tests that reject unauthenticated enumeration.

### 2) The popup enables “Send to Pi” even when the broker token is missing, so the user only fails after doing the page-selection work
- **Files:** `src/chrome/popup.ts:142-152`, `src/chrome/popup.ts:297-341`, `src/chrome/background.ts:456-469`, `src/chrome/background.ts:500-504`
- **Problem:** `listTargets` returns `tokenConfigured`, but popup send readiness only checks whether a target is selected. If a target exists and the token is absent, the popup still enables **Send to Pi**, starts the picker, lets the user choose content, and only then fails in the background with `No broker token configured in chrome.storage.local`.
- **Why it matters:** This is a high-friction failure mode in the main user flow. The user spends time selecting the DOM fragment and entering a comment before discovering the extension was never in a send-ready state. For an MVP that already requires manual token setup, the UI should fail early and clearly.
- **Review recommendation:** Treat missing token configuration as a first-class disabled state in the popup, with explicit guidance before the picker is launched; add popup tests that verify the button stays disabled and the status text explains the missing token case.

### 3) Broker startup is not resilient to the multi-session race it is supposed to support
- **Files:** `src/pi/browserConnectExtension.ts:468-503`
- **Problem:** When initial connect fails, the command tries to start the broker. If another Pi session wins that race and binds the fixed port first, this code path can receive `EADDRINUSE` from `startBrokerServer(...)` and fail the whole command instead of retrying a normal connect to the now-running broker.
- **Why it matters:** The architecture explicitly supports multiple Pi terminals joining the same broker. This race makes that workflow fragile exactly at startup time, when users are most likely to run `/browser-connect` in several terminals one after another.
- **Review recommendation:** Handle “broker already started elsewhere” as a recoverable branch: if broker startup fails due to address-in-use (or equivalent bind conflict), retry `connectAndActivate(...)` against the fixed port before surfacing an error. Add a focused regression test for that path.

## Minor

### 1) README is still partially in pre-implementation state
- **Files:** `README.md:22-24`, `README.md:73-75`
- **Problem:** The repository now contains a working MVP, but the README still says installation is “TBD — разработка в процессе” and leaves the license section as `TBD`.
- **Why it matters:** This undermines trust in the docs and makes the repo look less complete than the actual implementation. It also increases the chance that reviewers/operators miss the real setup path and current limitations.
- **Review recommendation:** Bring the top-level README fully in sync with the implemented MVP state and remove stale “in progress” placeholders.

# 4. Security and privacy audit

## What is good

- Broker binds to `127.0.0.1` only.
- Token creation/hardening is careful:
  - directory symlink rejection;
  - `O_NOFOLLOW` for secure token-file opens;
  - regular-file validation on opened descriptors;
  - permission tightening to `0700` for the directory and `0600` for the file.
- Logging generally avoids writing full selection payload contents.
- Selection payloads are size-bounded before they are formatted for Pi.
- Broker validates protocol envelopes and selection payload shape.

## Security issues

- **Important:** target enumeration is unauthenticated (`src/pi/broker.ts:407-415`), leaking sensitive local metadata.
- The protocol currently treats `client.hello` as optional in practice. That weakens the broker’s trust model and makes authentication easy to bypass for read-only operations.

## Residual privacy risks even after fixes

- The selected HTML/text is intentionally sent to Pi; users can still capture sensitive page data if they choose the wrong element.
- Diagnostics and log entries currently avoid payload bodies, but URL/title leakage is still possible in diagnostic messages, which is acceptable for MVP but worth documenting clearly.

# 5. Test and verification assessment

## What I verified

- `npm test` ✅
- `npm run typecheck` ✅
- `npm run build:chrome` ✅
- `pi --version` ✅ (`0.78.1`)
- temporary-`HOME` Pi smoke command: `HOME="$tmp_home" pi -p "/browser-connect review-smoke"` ✅

## Assessment

The automated suite is strong for this size of project. The broker tests are especially useful, and the popup/content-script tests cover more edge cases than I would normally expect in an MVP.

However, the current tests also reveal the main gaps:

- they explicitly encode the unauthenticated list-targets behavior as acceptable (`src/pi/broker.test.ts`);
- there is no regression test for the missing-token popup state, even though the background already exposes `tokenConfigured`;
- there is no test for the broker startup bind race in the multi-session flow;
- there is still no real browser-load / interactive extension verification in CI, so manifest/runtime correctness in an actual Chrome MV3 environment remains only partially proven.

Overall: **good unit/integration depth, but missing coverage exactly around the current blocking issues and real browser UX validation.**

# 6. Documentation assessment

## Accurate / useful

- README documents the manual broker-token bootstrap and the localhost-broker MVP limitation.
- Design and implementation-plan docs are detailed and largely align with the code structure.
- The README correctly explains that Pi answers are not streamed back into the browser in MVP.

## Inaccurate / incomplete

- Top-level README still contains stale “installation TBD / development in progress” language.
- There is no user-facing documentation for how to populate `chrome.storage.local["brokerToken"]` in practice, which matters because that is currently mandatory.
- The docs do not call out that the popup can appear ready even when token setup is incomplete; today that behavior is surprising.

# 7. Residual risks / follow-up recommendations

- Add authenticated gating for all broker reads before calling the MVP secure enough for routine use.
- Make popup readiness reflect the true send prerequisites, especially token availability.
- Harden broker startup for concurrent Pi sessions joining at nearly the same time.
- If this moves beyond MVP, add at least one real-browser smoke check (manual or automated) to validate MV3 loading, script injection, and popup-to-content-script flow.
- Consider documenting a minimal, supported way to set `brokerToken` for developers/users until a pairing UI exists.

# 8. Final verdict

**CHANGES REQUIRED**

The implementation is close and has many solid pieces, but the unauthenticated target enumeration and the late-failing no-token UX are important enough that I would not sign off on this branch yet. The broker startup race should also be fixed before considering the multi-session story reliable.
