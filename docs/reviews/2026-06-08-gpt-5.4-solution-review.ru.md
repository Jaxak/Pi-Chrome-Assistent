# 1. Краткое резюме

Ветка в целом в хорошем состоянии: кодовая база модульная, общая логика протокола/трункации хорошо выделена, работа с token-файлом на стороне Pi сделана необычно аккуратно для MVP, а автоматический набор тестов довольно широк для первого релиза. Я проверил `npm test`, `npm run typecheck`, `npm run build:chrome`, `pi --version`, а также smoke-запуск Pi с временным `HOME` и командой `/browser-connect review-smoke`; всё прошло успешно в этом окружении.

Тем не менее, я **не считаю**, что реализацию стоит оставлять как есть. Есть две блокирующие инженерные проблемы и ещё один важный пробел по устойчивости:

- broker позволяет перечислять target’ы без аутентификации по token, из-за чего любой локальный WebSocket-клиент может получить metadata вроде cwd/branch/pid/session;
- popup по-прежнему позволяет пользователю зайти в picker/send flow даже если `brokerToken` не настроен, из-за чего пользователь узнаёт об ошибке слишком поздно — уже после выбора DOM-фрагмента;
- запуск broker подвержен гонке: если две Pi-сессии почти одновременно пытаются стать broker, проигравшая может завершиться ошибкой вместо того, чтобы просто переподключиться.

# 2. Сильные стороны

- Чёткие модульные границы между shared protocol, Pi broker/client и Chrome UI/runtime.
- Хороший объём целевых автоматических тестов: protocol, truncation, broker integration, target client lifecycle, popup behavior, DOM picker heuristics и build output.
- Осознанная с точки зрения безопасности работа с token-файлом в `src/pi/browserConnectExtension.ts`.
- Логи намеренно не пишут полное содержимое selection payload по умолчанию.
- Поток DOM selection декомпозирован разумно: picker, overlay, toast и background messaging вынесены в отдельные модули.
- Smoke-path для Pi работает в этом окружении: `/browser-connect` поднимается, регистрируется и корректно завершается.

# 3. Замечания

## Критические

- Нет.

## Важные

### 1) Неаутентифицированные клиенты могут перечислять активные Pi target’ы и читать metadata workspace
- **Файлы:** `src/pi/broker.ts:394-416`, `src/chrome/background.ts:281-297`, `src/pi/broker.test.ts:238-274`
- **Проблема:** `client.hello` проверяет token, но broker не требует authenticated state для `client.listTargets`. Chrome background client тоже не отправляет `client.hello` перед listing. В текущем виде любой сокет, который может достучаться до `ws://127.0.0.1:17345`, может вызвать `client.listTargets` и получить alias, cwd, git branch, PID и session name.
- **Почему это важно:** Это реальная утечка privacy, а не косметический нюанс протокола. Браузерные страницы обычно могут открывать WebSocket на localhost, значит посещённый сайт может fingerprint’ить локальные проекты/ветки/процессы разработчика даже без shared token. Текущие тесты явно закрепляют это поведение как допустимое.
- **Рекомендация:** Требовать authenticated session state для всех client-операций, которые раскрывают данные broker, включая `client.listTargets`; заставить Chrome background выполнять handshake (или включать token в list-запрос), а текущий broker test, который утверждает, что listing работает “without prior hello”, заменить тестами, которые отклоняют неаутентифицированное перечисление.

### 2) Popup включает “Send to Pi” даже когда broker token отсутствует, поэтому пользователь узнаёт об ошибке только после выбора DOM
- **Файлы:** `src/chrome/popup.ts:142-152`, `src/chrome/popup.ts:297-341`, `src/chrome/background.ts:456-469`, `src/chrome/background.ts:500-504`
- **Проблема:** `listTargets` возвращает `tokenConfigured`, но готовность popup к отправке проверяет только наличие выбранного target. Если target существует, а token отсутствует, popup всё равно включает **Send to Pi**, запускает picker, позволяет выбрать контент, и только потом background падает с `No broker token configured in chrome.storage.local`.
- **Почему это важно:** Это высокофрикционный failure mode в основном пользовательском сценарии. Пользователь тратит время на выбор DOM-фрагмента и комментарий, прежде чем узнаёт, что extension вообще не была готова к отправке. Для MVP, где уже есть manual token setup, UI должен падать раньше и понятнее.
- **Рекомендация:** Считать отсутствие token полноценным disabled-state в popup и явно объяснять это до запуска picker; добавить popup tests, которые проверяют, что кнопка остаётся disabled и status text объясняет случай отсутствия token.

### 3) Запуск broker неустойчив к гонке multi-session сценария, который сам же должен поддерживать
- **Файлы:** `src/pi/browserConnectExtension.ts:468-503`
- **Проблема:** Когда initial connect не удался, команда пытается стартовать broker. Если другая Pi-сессия выигрывает эту гонку и раньше биндует фиксированный порт, этот код может получить `EADDRINUSE` от `startBrokerServer(...)` и завалить всю команду вместо того, чтобы просто повторно подключиться к уже поднявшемуся broker.
- **Почему это важно:** Архитектура явно поддерживает несколько Pi-терминалов, присоединяющихся к одному broker. Эта гонка делает workflow хрупким ровно в момент старта, когда пользователи чаще всего и запускают `/browser-connect` в нескольких терминалах подряд.
- **Рекомендация:** Обрабатывать “broker уже поднят где-то ещё” как recoverable branch: если запуск broker падает из-за занятого адреса (или аналогичного bind conflict), повторить `connectAndActivate(...)` к фиксированному порту перед тем, как показывать ошибку. Добавить целевой regression test на этот путь.

# 4. Аудит безопасности и privacy

## Что сделано хорошо

- Broker слушает только `127.0.0.1`.
- Создание и hardening token-файла сделаны аккуратно:
  - rejection symlink’ов для директории;
  - `O_NOFOLLOW` для безопасных opens token-файла;
  - regular-file validation на открытых descriptor’ах;
  - затягивание прав до `0700` для директории и `0600` для файла.
- Логи в целом избегают записи полного содержимого selection payload.
- Selection payload ограничивается по размеру перед форматированием для Pi.
- Broker валидирует protocol envelopes и shape selection payload.

## Проблемы безопасности

- **Важная:** перечисление target’ов не требует аутентификации (`src/pi/broker.ts:407-415`), из-за чего утекают чувствительные локальные metadata.
- Протокол сейчас фактически считает `client.hello` необязательным. Это ослабляет trust model broker и делает аутентификацию легко обходной для read-only операций.

## Остаточные privacy-риски даже после исправлений

- Выбранный HTML/text намеренно отправляется в Pi; пользователь всё ещё может случайно захватить чувствительные данные страницы, если выберет не тот элемент.
- Diagnostics и log entries сейчас избегают payload bodies, но URL/title всё ещё могут утекать в diagnostic messages. Для MVP это приемлемо, но стоит явно документировать.

# 5. Оценка тестов и верификации

## Что было проверено

- `npm test` ✅
- `npm run typecheck` ✅
- `npm run build:chrome` ✅
- `pi --version` ✅ (`0.78.1`)
- временный `HOME` + smoke-команда Pi: `HOME="$tmp_home" pi -p "/browser-connect review-smoke"` ✅

## Оценка

Автоматический набор тестов для проекта такого размера — сильный. Особенно полезны broker-тесты, а popup/content-script тесты покрывают больше edge cases, чем обычно ожидаешь от MVP.

Тем не менее, тесты также хорошо показывают текущие пробелы:

- они явно закрепляют неаутентифицированный `listTargets` как допустимое поведение (`src/pi/broker.test.ts`);
- нет regression test на состояние popup при отсутствии token, хотя background уже отдаёт `tokenConfigured`;
- нет теста на startup bind race для multi-session workflow;
- по-прежнему нет настоящей browser-load / interactive extension verification в CI, так что manifest/runtime correctness в реальном Chrome MV3 окружении доказаны лишь частично.

Итого: **хорошая глубина unit/integration покрытия, но не хватает тестов именно вокруг текущих блокирующих проблем и реальной browser UX validation.**

## Что корректно / полезно

- README документирует ручной bootstrap для broker-token и localhost-broker limitation в MVP.
- Дизайн и implementation-plan docs подробные и в целом согласованы с кодовой структурой.

## Что неточно / неполно

- Top-level README всё ещё содержит устаревшие формулировки вида “installation TBD / development in progress”.
- Нет user-facing документации о том, как именно practically заполнить `chrome.storage.local["brokerToken"]`, а это сейчас обязательно.
- Документация не акцентирует, что popup может выглядеть готовой к отправке даже если настройка token ещё не завершена.

# 7. Остаточные риски / рекомендации на будущее

- Добавить authenticated gating для всех broker read-операций, прежде чем считать MVP достаточно безопасным для обычного использования.
- Сделать popup readiness отражающим реальные prerequisites для отправки, особенно наличие token.
- Укрепить startup broker’а для почти одновременного подключения нескольких Pi-сессий.
- Если это перерастёт MVP, добавить хотя бы один real-browser smoke check (manual или automated) для проверки MV3 loading, script injection и popup-to-content-script flow.
- Рассмотреть документирование минимального поддерживаемого способа установки `brokerToken` для developer/user до появления pairing UI.
