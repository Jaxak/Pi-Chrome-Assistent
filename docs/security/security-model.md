# Модель безопасности

## Граница доверия

Chrome Assistent проектировался как **локальная** связка «браузер ↔ Pi» на одной машине.

Текущая модель доверия строится вокруг трёх решений:

1. broker слушает только `127.0.0.1`;
2. Pi-цели и браузеры используют **разные** токены доверия;
3. доверенные браузеры фиксируются в глобальном реестре Pi.

Это не zero-trust и не сетевой сервис общего назначения. Модель рассчитана на сценарий, где и браузер, и Pi запущены у одного локального пользователя.

## Localhost boundary

### Что уже ограничено

- broker bind'ится на `127.0.0.1`, а не на `0.0.0.0`;
- manifest Chrome-расширения разрешает только `http://127.0.0.1/*` и `ws://127.0.0.1/*`;
- удалённые хосты и внешняя сеть в текущей версии не участвуют.

### Чего это не даёт

- localhost не защищает от других локальных процессов того же пользователя;
- localhost не защищает, если browser token уже утёк в другой профиль или инструмент на той же машине.

## Разделение доверия

### Internal target token

Pi-часть использует внутренний token для `target.register`. Он хранится в:

```text
~/.pi/chrome-assistent/broker.token
```

Этот token не предназначен для ручного копирования в браузер.

### Browser token

Chrome-расширение использует отдельный browser token. Он хранится в `chrome.storage.local` браузера и показывается пользователю на вкладке **«Авторизация»**.

Когда пользователь выполняет `/chrome-assistent-auth`, Pi добавляет этот token в реестр доверенных браузеров:

```text
~/.pi/chrome-assistent/trusted-browsers.json
```

### Что это даёт

- браузер не получает внутренний token Pi;
- компрометация browser token не даёт автоматически зарегистрировать произвольную Pi-цель;
- можно перевыпустить browser token отдельно от внутреннего runtime Pi.

## Риск утечки метаданных

Доверенный браузер может видеть локальные метаданные Pi-целей:

- `cwd`
- `alias`
- `gitBranch`
- `pid`
- `sessionName`
- времена подключения и heartbeat

Поэтому утечка browser token означает и утечку локальных рабочих метаданных.

## Риск утечки содержимого страницы

При `client.sendSelection` в Pi уходит:

- URL страницы;
- title;
- CSS selector;
- выбранный текст;
- HTML-фрагмент;
- необязательный комментарий.

Поэтому пользователь не должен считать Chrome Assistent каналом с автоматической классификацией или редактированием чувствительных данных.

## Текущие смягчающие меры

В коде уже есть несколько полезных защит:

- loopback-only binding (`127.0.0.1`);
- отдельный token для browser auth и отдельный token для Pi target registration;
- явный доверенный реестр браузеров в Pi;
- `client.listTargets` требует успешный `client.hello`;
- ошибки аутентификации закрывают socket;
- heartbeat и stale cleanup удаляют мёртвые цели;
- валидация protocol envelope и payload;
- ограничение размера текста и HTML при формировании сообщения для Pi.

## Остаточные риски

### 1. Browser token живёт в профиле браузера

Если браузерным профилем пользуется кто-то ещё, доверенный token тоже может стать общим.

### 2. Один фиксированный порт

Даже после разделения доверия broker всё ещё использует фиксированный `17345`.

### 3. Нет шифрования поверх localhost

Трафик идёт по `ws://`, а не по `wss://`. Для loopback-сценария это принято как упрощение.

### 4. Нет отдельного фонового сервиса broker

Жизненный цикл broker по-прежнему зависит от живой Pi-сессии.

## Что важно пользователю прямо сейчас

- не делитесь browser token из вкладки **«Авторизация»** с другими людьми;
- если перевыпустили token, повторите `/chrome-assistent-auth`;
- перед отправкой проверяйте, что DOM picker выделил именно тот фрагмент, который вы хотите передать в Pi;
- если работаете в чувствительном браузерном профиле, удаляйте token, когда он больше не нужен.
