# Модель безопасности

## Граница доверия в текущем MVP

Browser Connect проектировался как **локальная** связка «браузер ↔ Pi» на одной машине.

Текущая защитная граница строится вокруг двух решений:

1. broker слушает только `127.0.0.1`;
2. доступ к broker защищён общим токеном.

Это не zero-trust и не сетевой сервис общего назначения. Модель рассчитана на сценарий, где и браузер, и Pi запущены у одного локального пользователя.

## Граница localhost

### Что уже ограничено

- broker bind'ится на `127.0.0.1`, а не на `0.0.0.0`;
- manifest Chrome-расширения разрешает только `http://127.0.0.1/*` и `ws://127.0.0.1/*`;
- удалённые хосты и внешняя сеть в MVP не участвуют.

### Что это даёт

- удалённый узел не может напрямую подключиться к broker по сети;
- WebSocket-трафик не выходит за пределы loopback-интерфейса.

### Чего это не даёт

- localhost не защищает от других локальных процессов того же пользователя;
- localhost не защищает, если токен утёк в другой браузерный профиль, локальный инструмент или логи.

## Аутентификация токеном

### Как токен создаётся и хранится

Pi-часть использует файл:

```text
.pi/browser-connect.token
```

Токен создаётся как UUID и затем переиспользуется. Код дополнительно:

- создаёт каталог `.pi` с правами `0700`;
- защищает файл токена правами `0600`;
- запрещает симлинки для каталога и файла токена.

### Где токен ещё хранится

Для работы браузера пользователь вручную переносит токен в:

```text
chrome.storage.local["brokerToken"]
```

Это удобно для MVP, но означает, что токен хранится ещё и в профиле браузера.

### Как токен используется

- `client.hello` аутентифицирует browser client перед `client.listTargets`;
- `client.sendSelection` несёт токен в payload;
- `target.register` требует тот же токен для регистрации Pi-цели.

При неверном токене broker возвращает `Invalid token` и закрывает socket.

## Риск утечки метаданных

Даже без отправки реального контента authenticated client может узнать о локальных Pi-целях:

- `cwd`
- `alias`
- `gitBranch`
- `pid`
- `sessionName`
- времена подключения и heartbeat

Это означает, что **утечка токена = утечка локальных рабочих метаданных**. Для MVP это считается допустимым риском, потому что транспорт ограничен localhost, но риск всё равно нужно понимать.

## Риск утечки содержимого страницы

При `client.sendSelection` в Pi уходит:

- URL страницы;
- title;
- CSS selector;
- выбранный текст;
- HTML-фрагмент;
- необязательный пользовательский комментарий.

Поэтому пользователь не должен считать Browser Connect каналом для чувствительных данных с автоматической классификацией или redaction. Всё, что выбрано в DOM picker, может попасть в контекст Pi-сессии.

## Текущие смягчающие меры

В коде уже есть несколько полезных защит:

- loopback-only binding (`127.0.0.1`);
- обязательный токен для target registration и browser auth;
- `client.listTargets` теперь требует предварительный `client.hello`;
- ошибки аутентификации закрывают socket;
- heartbeat и stale cleanup удаляют мёртвые цели;
- валидация структуры protocol envelope и payload;
- ограничение размера текста и HTML при формировании сообщения для Pi;
- Pi-логи не записывают полный HTML/текст выделения как штатный лог-событие.

## Остаточные риски MVP

### 1. Ручное provision токена

Пока токен копируется вручную, возможны:

- хранение токена в истории DevTools;
- случайная подстановка токена не от того проекта;
- длительное сохранение токена в профиле браузера.

### 2. Один фиксированный порт

Брокер всегда использует `17345`, а токен берётся из `.pi/browser-connect.token` текущего проекта. Это создаёт риск конфликтов и путаницы между несколькими независимыми проектами на одной машине.

### 3. Нет шифрования поверх localhost

Трафик идёт по обычному `ws://`, а не по `wss://`. Для loopback-сценария это принято как упрощение MVP, но это не защита от полностью скомпрометированной локальной машины.

### 4. Нет явного UI-подтверждения на каждую регистрацию браузера

Если токен уже известен браузерному расширению, оно может запрашивать список целей и отправлять выделения без дополнительного подтверждения на стороне Pi.

### 5. Нет детальной изоляции между проектами

Архитектура не обеспечивает прозрачную мультипроектную работу с разными токенами на одном хосте.

## Что важно пользователю прямо сейчас

- храните `.pi/browser-connect.token` как локальный секрет;
- не оставляйте лишний `brokerToken` в браузерных профилях, которыми пользуются другие люди;
- перед отправкой проверяйте, что DOM picker выделил именно тот фрагмент, который вы хотите передать в Pi;
- если работаете с несколькими проектами, сознательно переключайте `brokerToken` и активную Pi-сессию.
